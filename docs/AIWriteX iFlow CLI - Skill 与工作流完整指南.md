## 概述

AIWriteX 的 iFlow CLI 系统是一个**结构化工作流框架**，通过 **Skill（技能）** 和 **Command（命令）** 帮助开发者：

- 🎯 **标准化开发流程** - 每个任务都有明确的步骤和规范
- 🚀 **提升代码质量** - 通过审查、测试、验证等机制
- 📈 **提高开发效率** - 避免重复造轮子，复用最佳实践
- 🐛 **减少错误** - 系统化的调试和验证流程

### 核心哲学

> **"纪律产生自由"** - 通过严格遵循经过验证的工作流程，开发者可以更专注于创造性的工作，而不必担心遗漏关键步骤。

## 什么是 Skill

**Skill（技能）** 是预定义的工作流程模板，用于指导特定类型的任务。每个 Skill 包含：

- 📋 **详细的步骤说明** - 告诉你要做什么
- ⚠️ **危险信号提醒** - 告诉你什么不能做
- 🔗 **集成点** - 与其他 Skill 的配合方式
- ✅ **完成标准** - 何时算完成任务

### Skill 的两大类型

| 类型 | 说明 | 示例 |
|------|------|------|
| **严格型** | 必须严格遵循，不能偏离 | TDD（测试驱动开发）、调试 |
| **灵活型** | 根据上下文调整原则 | 头脑风暴、计划编写 |

### 如何使用 Skill

在 iFlow CLI 中，使用 `Skill` 工具调用：

```bash
# 调用特定 Skill
Skill: "brainstorming"

# 或者在对话中自动触发
# 当识别到适用场景时，系统会自动建议使用相应 Skill
```

## 什么是 iFlow 工作流

**iFlow 工作流** 是 AIWriteX 提供的**长期运行项目管理框架**，特别适合需要多个会话完成的复杂项目。

### 核心组件

```
.iflow/
├── commands/          # 命令定义
│   ├── long-running-init.md
│   ├── long-running-coding.md
│   ├── long-running-status.md
│   └── ...
├── skills/            # 技能定义
│   ├── brainstorming/
│   ├── writing-plans/
│   ├── executing-plans/
│   └── ...
├── progress.md        # 项目进度追踪
└── features.json      # 功能需求列表
```

### 三种执行模式

| 模式 | 适用场景 | 特点 |
|------|----------|------|
| **单会话执行** | 简单任务（< 1小时） | 快速、直接 |
| **子代理驱动** | 中等复杂度任务 | 同一会话，自动审查 |
| **并行会话** | 复杂长期项目 | 独立工作树，批量执行 |

## 14 个核心 Skill 详解

### 1. using-superpowers ⚡ [严格型]

**用途**: 所有对话的入口点，确立如何查找和使用技能

**核心价值**:
- 确保不会遗漏适用的 Skill
- 建立"先检查 Skill，再行动"的肌肉记忆
- 强制规范化工作流程

**使用时机**: **每次对话开始时**

**效果**: 
- ✅ 避免"我觉得这很简单不需要规范"的陷阱
- ✅ 自动识别并建议使用适用的 Skill
- ✅ 确保工作质量的一致性

---

### 2. brainstorming 🧠 [灵活型]

**用途**: 将想法转化为完整的设计和规范

**适用场景**:
- 创建新功能
- 构建新组件
- 添加新功能
- 修改现有行为

**工作流程**:
```
1. 探索项目背景 → 2. 提出澄清问题 → 3. 提出2-3种方法 
→ 4. 展示设计 → 5. 编写设计文档 → 6. 过渡到实施
```

**核心价值**:
- 🎯 **防止盲目开发** - 先理解需求，再动手编码
- 💡 **探索多种方案** - 避免陷入第一种想法
- 📝 **文档化设计** - 留存决策依据，便于团队协作
- ✅ **用户确认** - 确保方向正确

**输出产物**:
- 设计文档: `docs/plans/YYYY-MM-DD-<topic>-design.md`
- 明确的实施方向

---

### 3. writing-plans 📝 [严格型]

**用途**: 创建详细的实施计划

**适用场景**:
- 已有设计方案，需要具体实施步骤
- 多步骤任务需要分解
- 需要指导他人（或未来的自己）执行

**任务粒度**:
每个任务应该是**2-5分钟**能完成的操作：
```
❌ "实现用户系统" (太大)
✅ "创建 User 模型" (合适)
✅ "编写用户注册测试" (合适)
```

**计划模板**:
```markdown
# [功能] 实施计划

**目标:** [一句话描述]
**架构:** [2-3句话]
**技术栈:** [关键技术]

---

### 任务 1: [名称]

**文件:**
- 创建: `path/to/file.py`
- 修改: `path/to/existing.py:123-145`
- 测试: `tests/path/test.py`

**步骤:**
1. 编写失败的测试
2. 运行验证失败
3. 编写最小实现
4. 运行验证通过
5. 提交
```

**核心价值**:
- 📋 **可执行性** - 计划具体到每一步命令
- 🎯 **完整性** - 包含文件路径、代码、测试、预期输出
- 🔄 **DRY原则** - 不要重复造轮子
- 🚫 **YAGNI原则** - 不做不需要的功能

---

### 4. executing-plans 🚀 [严格型]

**用途**: 在单独会话中执行已编写的计划

**适用场景**:
- 计划已经编写完成
- 需要在隔离环境中执行
- 需要设置审查检查点

**执行流程**:
```
1. 加载并审查计划
2. 批量执行任务（默认前3个）
3. 报告进度，等待反馈
4. 根据反馈调整
5. 继续下一批
6. 完成开发
```

**核心价值**:
- 🏗️ **隔离执行** - 不污染当前会话
- 📊 **进度可控** - 批次执行，便于审查
- 🔄 **反馈循环** - 及时获得反馈并调整
- 🛑 **风险控制** - 遇到阻碍立即停止，不强行突破

---

### 5. subagent-driven-development 🤖 [严格型]

**用途**: 在当前会话中通过子代理执行计划

**vs. executing-plans 的区别**:

| 维度 | subagent-driven | executing-plans |
|------|-----------------|-----------------|
| 会话 | 同一会话 | 并行新会话 |
| 审查 | 每任务两阶段审查 | 批次间审查 |
| 速度 | 更快（无需等待） | 较慢（需人工介入） |
| 上下文 | 独立（无混淆） | 完全隔离 |

**执行流程**:
```
任务 N:
  ↓
[分发实施子代理] → 实现、测试、提交、自我审查
  ↓
[分发规范审查子代理] → 确认符合规范？
  ↓ 否
[实施者修复] → 重新审查
  ↓ 是
[分发代码质量审查子代理] → 代码质量审查
  ↓ 否
[实施者修复] → 重新审查
  ↓ 是
标记完成 → 下一任务
```

**核心价值**:
- 🤖 **自动化审查** - 规范审查 + 代码质量审查
- 🔄 **快速迭代** - 任务之间无需人工等待
- ✅ **质量保证** - 两阶段审查确保质量
- 🎯 **精准执行** - 子代理专注单一任务，上下文清晰

---

### 6. test-driven-development 🧪 [严格型]

**用途**: 测试驱动开发

**核心循环**:
```
1. 编写失败的测试
2. 运行测试（确认失败）
3. 编写最少代码使测试通过
4. 运行测试（确认通过）
5. 重构（保持测试通过）
6. 提交
```

**测试粒度**:
- 每个测试只验证一个概念
- 测试名应该描述行为：`test_[行为]_[条件]_[结果]`

**核心价值**:
- ✅ **高测试覆盖率** - 每个功能都有测试
- 🐛 **早发现错误** - 在编码时发现而非运行时发现
- 📚 **活文档** - 测试即文档，描述系统行为
- 🔄 **安全重构** - 有测试保护，敢重构

---

### 7. systematic-debugging 🔍 [严格型]

**用途**: 系统化调试

**调试五步法**:
```
1. 收集证据 - 日志、错误信息、复现步骤
2. 形成假设 - 基于证据提出可能原因
3. 预测结果 - 如果假设正确，会发生什么
4. 验证假设 - 设计实验验证
5. 修正问题 - 修复并验证
```

**核心价值**:
- 🎯 **避免随机尝试** - 系统化方法，不猜不蒙
- 📊 **证据驱动** - 基于事实而非直觉
- 🔄 **快速收敛** - 通过假设-验证循环快速定位问题
- 📝 **知识积累** - 调试过程就是学习过程

---

### 8. requesting-code-review 👥 [灵活型]

**用途**: 请求代码审查

**审查清单**:
- [ ] 代码是否符合设计规范
- [ ] 是否有未处理的边界情况
- [ ] 测试是否充分
- [ ] 是否有代码异味（重复、过长函数等）
- [ ] 文档是否更新

**核心价值**:
- 👀 **第二双眼睛** - 发现作者遗漏的问题
- 📚 **知识共享** - 团队成员了解代码变更
- 🎓 **学习机会** - 通过反馈提升技能
- ✅ **质量把关** - 防止低质量代码进入主干

---

### 9. receiving-code-review 💬 [灵活型]

**用途**: 接收代码审查反馈

**原则**:
- 不要防御性反应
- 每个建议都要考虑
- 不懂就问，不要猜测
- 区分"必须修复"和"建议优化"

**核心价值**:
- 🎯 **技术成长** - 从反馈中学习
- 🤝 **团队协作** - 建立积极的反馈文化
- ✅ **代码质量** - 持续改进

### 10. using-git-worktrees 🌲 [严格型]

**用途**: 使用 Git Worktree 进行并行开发

**什么是 Worktree**:
一个 Git 仓库可以有多个工作目录，每个目录对应不同的分支，彼此独立。

**使用场景**:
- 同时处理多个功能分支
- 长期运行的功能开发
- 避免频繁切换分支的上下文丢失

**核心价值**:
- 🔄 **并行开发** - 多个分支同时推进
- 🧹 **环境隔离** - 互不干扰
- ⚡ **快速切换** - 无需 stash/pop，直接切换目录
- 📁 **清晰组织** - 每个功能一个目录

## 使用场景与最佳实践

### 场景 1: 开发新功能

**完整工作流**:
```
1. Skill: brainstorming
   → 输出: 设计文档

2. Skill: writing-plans
   → 输出: 实施计划

3. 选择执行方式:
   a) Skill: subagent-driven-development (同一会话)
   b) Command: /execute-plan (并行会话)

4. Skill: requesting-code-review
   → 代码审查

5. Skill: finishing-a-development-branch
   → 完成合并
```

### 场景 2: 修复 Bug

**工作流**:
```
1. Skill: systematic-debugging
   → 定位问题

2. Skill: writing-plans (如需要)
   → 编写修复计划

3. 实施修复

4. Skill: verification-before-completion
   → 验证修复

5. Skill: requesting-code-review
   → 审查
```

### 场景 3: 长期项目

**工作流**:
```
1. Command: /long-running-init
   → 初始化项目

2. 循环直到完成:
   Command: /long-running-coding
   → 开发功能

3. Command: /long-running-status
   → 查看进度
```

### 真实案例

3. `subagent-driven-development` - 并行开发 5 个核心模块

- ✅ 所有测试通过
- ✅ 零回归错误
- ✅ 文档完整

#### 案例 2: 新功能开发

**背景**: 添加实时协作功能

**使用的工作流**:
1. `brainstorming` - 3种方案对比，选择最优
